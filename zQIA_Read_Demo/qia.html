Mathematical Foundation — QUBO Formulation

Explain it like I am new: Think of this as a smart "arrange classes" machine. It starts with a rough schedule, checks what is bad, then keeps making better swaps until the schedule is clean and practical.

The scheduling problem is encoded as a Quadratic Unconstrained Binary Optimization (QUBO) — we seek an assignment that minimizes a global energy (cost) function.

x* = arg min E(x) = arg min [ λ·H(x) + Σ wᵢ·Sᵢ(x) ]

Variables

  • x — assignment vector: each section s mapped to (room r, day d, time t)
  • E(x) — total energy of assignment x
  • λ = 1,000,000 — hard constraint penalty
  • wᵢ ∈ [10, 500] — soft constraint weights

Objective

  • H(x) — sum of all hard constraint violations
  • Sᵢ(x) — soft constraint penalty terms
  • Goal: drive E(x*) → 0
  • Hard violations: E(x) ≥ 1,000,000 (infeasible)

Main Scheduler — Simulated Annealing Core

Metropolis Acceptance Criterion
In plain words: The system tries small changes. If a change improves the schedule, it keeps it. If a change is worse, it might still keep it early on so it can explore and avoid getting stuck.
// Core optimization loop — mathematical form: Given: x₀ ← GreedyConstruct(S, R, T, D) T₀ ← initial temperature, α ← cooling rate ∈ (0,1) At each iteration k: x' ← GetNeighbor(xₖ) // small perturbation Δ ← E(x') - E(xₖ) // energy delta // Metropolis Acceptance: P(accept) = { 1 if Δ < 0 { exp(−Δ / Tₖ) if Δ ≥ 0 // Adaptive Cooling: Tₖ₊₁ = max(Tₖ × α, T_min) // Quantum Tunneling trigger: IF stagnation_count > 50 THEN QuantumTunnel(xₖ, Tₖ) Termination: E(x*) = 0 // perfect schedule E(x*) < 1000 AND k > 100 // good enough No improvement for 500 iters // exhausted

Acceptance Probability vs ΔE at Different Temperatures

Why exp(−Δ/T)?

When T is high, even bad moves (Δ > 0) are accepted with high probability — the algorithm explores freely. As T decreases, only improvements are accepted — the algorithm exploits promising regions. This prevents getting stuck in local minima.

Cooling Schedule

Geometric cooling: Tₖ = T₀ × αᵏ

Typical values: T₀ = 1000, α = 0.995, T_min = 0.1. After 1000 iterations, T drops to ≈ 0.67% of initial, making acceptance of worse solutions extremely rare.

Greedy Construction — Initial Solution

Priority-Ordered Placement
In plain words: The algorithm first places the hardest classes (fixed, lab, big sections). Easy classes are placed later. This gives a much better starting point before deep optimization.
// Formally: a prioritized greedy heuristic Define priority function π(s) for section s: π(s) = { 4 if s is pinned (admin-fixed) { 3 if s is a lab session { 2 if student_count(s) is large { 1 if weekly_hours(s) is large { 0 otherwise Sort: S_sorted = sort(S, key=π, descending) For each section s ∈ S_sorted: best_slot* = arg min { fit_cost(r, d, t) } (r,d,t) ∈ compatible_rooms[s] × D × T subject to: no_conflict(s, r, d, t) = TRUE fit_cost(r, d, t) = capacity_waste(r, s) + time_preference(t) // Non-consecutive day constraint: IF ∃ session s' of same subject on day d±1 THEN skip d

Session Splitting Rule

A 3-hr lecture → 1 session. A 5-hr class → 2 sessions (e.g., 3hr + 2hr). This prevents scheduling impossibly long single-day blocks and spreads cognitive load.

Non-Consecutive Day Rule

If subject A meets on Monday, it cannot meet on Tuesday (adjacent). This forces spacing, reducing student fatigue. Mathematically: |d₁ − d₂| ≥ 2 for same subject sessions.

Energy Function — QUBO Cost Calculation

Objective Function E(x)
In plain words: "Energy" is just a score. Bad schedule issues add points. Lower score is better. Hard rule breaks add huge points so the optimizer avoids them first.
// Full energy function: E(x) = λ · H(x) + Σᵢ wᵢ · Sᵢ(x) // ── Hard Constraints H(x): penalty = λ = 1,000,000 ────── H(x) = Σ [ room_double_book(r,d,t) // same room, 2+ classes + teacher_double_book(p,d,t) // teacher in 2 places + teacher_teleport(p,d,t,t+1) // diff building, consecutive + online_room_mismatch(c) // online class in phys. room + outside_hours(c) // before 7AM or after 9PM + capacity_exceeded(c) // students > 110% capacity + lab_in_wrong_room(c) // lab ↔ lecture room swap + strict_lunch_overlap(c) ] // overlaps lunch (strict mode) // ── Soft Constraints Sᵢ(x): weighted penalties ─────────── S(x) = 50 · room_type_mismatch(c) + 15 · waste_ratio(c) // (capacity - students)/capacity + 500 · lunch_overlap_flex(c) + 80 · extra_daily_hours(p) + 150 · late_lecture(c) + 100 · lec_lab_diff_days(c) + 200 · teacher_gap_3h(p) - 10 · accessible_room_bonus(c) // reward good placement

Quantum Tunneling — Escaping Local Minima

Large-Step Perturbation
In plain words: If the optimizer stops improving for too long, it makes a big jump to a new arrangement. This is the "quantum tunneling" idea.
// Triggered when: stagnation_count > 50 iterations // Makes a LARGE structural change — unlike SA's small steps Strategy ← Uniform({block_swap, random_replacement, online_shift}) IF block_swap: dept ← random department d ∈ Departments day₁, day₂ ← two distinct days (same college) x' = x // swap ALL classes of dept between day₁ ↔ day₂ IF random_replacement: s ← random section s ∈ S x' = x \ {s → (r, d, t)} // remove from current slot x' = x' ∪ {s → (r', d', t')} where // place anywhere valid (r', d', t') = arg min fit_cost IF online_shift: s ← random non-lab section x' = move s between physical ↔ online day // Metaphor: quantum particle tunnels through energy barrier // instead of climbing over it (SA) — jumps past local minima

Energy Landscape — SA vs Quantum Tunneling

🔵 SA gradually climbs out of local minima by accepting worse solutions with probability exp(−Δ/T)  |  ⚛️ Quantum Tunneling jumps past the barrier entirely

Why "Quantum"?

In quantum physics, particles can tunnel through energy barriers they classically couldn't overcome. Here we simulate this: when SA gets stuck (stagnation > 50), we make a large structural jump — "tunneling" past the energy barrier — rather than slowly climbing over it.

Stagnation Trigger

If no improvement in E(x) for 50+ consecutive iterations, standard SA is stuck in a local minimum. Quantum tunneling provides a controlled escape by perturbing the global structure (entire department days), not just single assignments.

Neighbor Generation — Local Perturbation

3-Move Operator
In plain words: Each step edits only one thing: room, day, or time. Tiny moves let the algorithm improve gradually without destroying the full schedule.
// Neighborhood function N(x): generates candidate x' near x Select: a ← random assignment (a = s → (r, d, t)), a ∉ Pinned Move type ← Uniform({change_room, change_day, change_time}) change_room: r' ← r' ∈ compatible_rooms[s], r' ≠ r x' = x \ {s→(r,d,t)} ∪ {s→(r',d,t)} // Valid if: room_available(r', d, t) = TRUE change_day: d' ← d' ∈ D, d' ≠ d, d' not restricted for labs, |d' - d_other_session| ≥ 2 // non-consecutive rule x' = x \ {s→(r,d,t)} ∪ {s→(r,d',t)} // Valid if: no teacher conflict on (d', t) change_time: t' ← t' ∈ T, t' ≠ t, t' ∉ lunch_slots (strict mode) x' = x \ {s→(r,d,t)} ∪ {s→(r,d,t')} // Valid if: no room or teacher conflict on (d, t') IF no valid move found → RETURN NULL // skip this iteration

The Three Move Types

🏠

Change Room

Keep same day and time. Move to a different compatible room. Fixes room-type mismatches and capacity waste.

📅

Change Day

Keep same room and time slot. Move to a different active day. Respects non-consecutive and online-day rules.

🕐

Change Time

Keep same room and day. Slide to a different time slot. Avoids lunch conflicts and late-day penalties.
